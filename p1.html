Project 1: AI Content Summarizer SaaS

An in-depth implementation guide for the senior engineer roadmap.

Architecture & Data Flow

Project Goal: To build a high-performance backend service that accepts text or a URL, uses an external AI service to generate a summary and keywords, and persists the results in a relational database. The entire application will be containerized for reproducibility.

Core Technologies:

  • Backend: Python with FastAPI
  • Database: PostgreSQL
  • ORM: SQLModel
  • AI Service: OpenAI API
  • Containerization: Docker & Docker Compose

Backend Implementation Strategy (FastAPI)

1

Data Modeling with Pydantic & SQLModel

The key is to define your data structures once and reuse them for API validation, serialization, and database schema definition. SQLModel, built on Pydantic, makes this seamless.

You'll create a base model with shared fields, then extend it for different contexts (API input, DB storage, API output).

# Snippet: sql_models.py
from typing import List, Optional
from sqlmodel import Field, SQLModel

class SummaryRequest(SQLModel):
    text: str
    # ... other input fields

class SummaryResult(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    original_text: str
    summary: str
    keywords: List[str] = Field(..., sa_column=...) # Use JSON type for list
2

Managing Database Sessions with Dependency Injection

Avoid managing database connections directly in your endpoint logic. Use FastAPI's `Depends` system to create a clean, reusable dependency that provides a database session and ensures it's closed after the request is finished, even if an error occurs.

# Snippet: database.py
from sqlmodel import Session, create_engine

def get_session():
    with Session(engine) as session:
        yield session

# Snippet: main.py
from fastapi import Depends
from sqlmodel import Session

@app.post("/summarize")
def create_summary(..., session: Session = Depends(get_session)):
    # session is now available here
    ...

Integrating the AI Service Asynchronously

External API calls are I/O-bound and perfect for `async/await`. Create a dedicated service module to handle the interaction with the OpenAI API. This isolates the logic and makes it testable. Use environment variables (`os.getenv()`) to manage your API key securelyâ€”never hard-code it.

# Snippet: ai_service.py
import openai
import os

openai.api_key = os.getenv("OPENAI_API_KEY")

async def get_ai_summary(text: str):
    response = await openai.Completion.acreate(...) # Use async method
    return response.choices[0].text.strip()

Structuring the Main Endpoint Logic

The main endpoint should act as a coordinator. It validates input, gets a DB session, calls the AI service, saves the result, and returns the response. This keeps the endpoint clean and readable.

# Snippet: main.py
@app.post("/summarize", response_model=SummaryResult)
async def create_summary(request: SummaryRequest, ...):
    # 1. Call the AI service
    summary_text = await ai_service.get_ai_summary(request.text)

Containerizing with Docker Compose

For a robust development and deployment workflow, use Docker Compose to manage the FastAPI application and the PostgreSQL database as connected services. This ensures your environment is consistent everywhere.

Key `docker-compose.yml` concepts: Define two `services`: one for your `api` (built from your `Dockerfile`) and one for the `db` (using the official `postgres` image). Use `volumes` to persist database data and `environment` to pass connection details and API keys securely.
